SECTION 1: PROJECT OVERVIEW AND VISION
─────────────────────────────────────────────────────────────────────────────

1.1 Vision Statement
────────────────────
The Data Science Learning Handbook is a standalone, publishable-quality
educational resource for data science practitioners working on federal
government platforms. It is not a textbook. It is not a blog post. It is the
book a senior practitioner would hand to someone on their first day on a
DoD contract and say: "Read this. Everything you need is in here."

The handbook teaches the full data science lifecycle — from initial environment
setup and data wrangling through machine learning, MLOps, visualization,
deployment, governance, and the current frontier of generative AI — through the
lens of the five platforms where federal data scientists actually work. Every
chapter grounds its content in real operational scenarios, government compliance
requirements, and platform-specific implementation patterns.

The content must not sound AI-generated. It must read like a senior
practitioner — confident, specific, direct, and willing to state opinions.
The voice reference is Amyn Porbanderwala's "Shrink-Wrap It: The GovCon
Productization Playbook."

1.2 What This Resource Is
─────────────────────────
- A 13-chapter handbook covering the complete data science lifecycle
- Platform-specific coverage of Advana, Qlik, Databricks, Navy Jupiter,
  and Palantir AIP/Foundry
- Working Python code examples for every major concept
- Hands-on exercises with solutions
- Platform comparison guides showing how the same task is accomplished across
  all five environments
- A reference that practitioners can use on active projects — not just study
  for a certification

1.3 What This Resource Is Not
──────────────────────────────
- Not a beginner's "learn to code" tutorial — the target audience already
  programs; they need to know how to apply data science specifically in
  federal environments
- Not a platform marketing document — the handbook takes positions on where
  each platform excels and where it falls short
- Not a classified document — all content is based on publicly available
  information about these platforms
- Not a finished product at the start of execution — this PRD defines what
  "done" means so that task generation, QA gates, and final sign-off have a
  clear standard to measure against

1.4 Success in One Sentence
────────────────────────────
A data scientist starting a new DoD contract reads this handbook and is
immediately effective — they know how to access data, build and deploy models,
satisfy compliance requirements, and get their work into production on the
specific platform their customer is running.

SECTION 2: TARGET AUDIENCE AND USER PERSONAS

2.1 Primary Audience
─────────────────────
Data science practitioners (contractors and government civilians) working on
federal and DoD projects. They have the foundational skills — they can write
Python, they understand what a machine learning model is — but they've never
worked inside a federal platform environment, or they've worked on one platform
and need to get up to speed on another.

for "how do I run a machine learning model on Advana under DISA IL4 constraints
while maintaining proper audit trails."

CHAPTER 01: Introduction to Data Science in Government
Directory: chapters/01-introduction/
──────────────────────────────────────────────────────────────
Scope:
This chapter orients the practitioner to the federal data science environment
specifically. It does not re-explain what data science is — it explains how
data science is different when your customer is a government agency, your data
is classified or sensitive, your platform is government-authorized, and your
output has to survive a 90-day Authority to Operate process.

Topics:
- The federal data landscape: how government data is structured, governed, and
  classified (FOUO, CUI, Controlled Unclassified Information, classified tiers)
- The five platforms covered in this handbook and why they matter: Advana
  (DoD's enterprise analytics hub), Qlik (BI layer on Advana and standalone),
  Databricks (lakehouse for heavy ML workloads), Navy Jupiter (DON enterprise
  data environment), Palantir AIP/Foundry (ontology-based AI platform)
- The data science lifecycle in a federal context: where CRISP-DM meets ATO
  requirements, compliance checkpoints, and acquisition cycles
- Tools and languages: Python (required), R (preferred), SQL, and when to use
  which on which platform
- Environment access patterns: CAC/PIV authentication, NIPRNET vs. SIPRNET,
  cloud vs. on-premises constraints per platform
- The DoD data strategy and CDAO context: why these platforms exist and what
  policy drives them (DoD Data Strategy 2020, EO 13960 on AI in government,
  CDAO charter)
- How to think about security from Day 1 — not as an afterthought

Learning Objectives:
After this chapter, the practitioner can explain what makes federal data science
distinct from commercial practice, name the five platforms and their primary
use cases, and knows which authentication method and network they'll need for
each platform.

Platform Coverage: All five platforms introduced at overview level.
Code Examples: Environment setup and CAC-aware authentication skeleton for each
platform. Python environment configuration with DoD proxy settings.

Scope:
Not a Python tutorial. This chapter covers the specific Python and R patterns,

  Qlik Engine API, Palantir Foundry API
- R in government contexts: when R is available, typical use cases (statistical
  reporting, actuarial work), integration with Python workflows
- Code quality standards for government deliverables: type hints, docstrings,
  reproducibility, version control in CAC-authenticated git environments
- Jupyter notebooks on government platforms: differences between local Jupyter,
  Databricks notebooks, and Palantir Foundry Code Workbook

Learning Objectives:
After this chapter, the practitioner knows which libraries and patterns to reach
for in a federal environment, can configure a working Python environment on a
government system, and can write production-quality code that meets government
delivery standards.

Platform Coverage: All five platforms for environment setup; Databricks and
Palantir in depth for notebook-based workflows.
Code Examples: pip/conda environment with proxy configuration; Databricks
cluster setup script; Palantir Code Workbook initialization pattern.

CHAPTER 03: Data Acquisition in Federal Environments
Directory: chapters/03-data-acquisition/
──────────────────────────────────────────────────────────────
Scope:
Getting data in a federal environment is harder than in commercial work — not
because the technology is harder, but because the access control, approval, and
governance layers are thicker. This chapter is about getting the data you need
while staying inside the compliance boundary.

Topics:
- Federal data sources: USASpending.gov, Data.gov, FPDS-NG (Federal Procurement
  Data System), FRED (economic data), agency-specific data portals — how to
  access each programmatically
- Advana data catalog: how Advana ingests from 1,200+ DoD business systems;
  how to discover what datasets exist; how to request access
- Navy Jupiter data discovery: finding datasets, understanding the metadata
  layer, submitting data access requests
- Databricks data sources: Unity Catalog metadata browsing, Delta Lake ingestion
  patterns, external data connector configuration in government cloud
- Palantir Foundry: the Ontology model for data discovery; Dataset Browser;
  connecting to external systems via Foundry connectors
- Qlik data connections: loading data from SQL databases, REST APIs, and file
  sources via Qlik Data Load Editor and QVD optimization
- API-based data acquisition: handling CAC/certificate authentication in Python
  requests; rate limiting; pagination patterns for large government datasets
- Streaming and real-time ingestion patterns available per platform
- Data governance before you touch the data: data use agreements, Memoranda of
  Understanding (MOUs), read vs. write access, OPSEC considerations

Learning Objectives:
After this chapter, the practitioner can discover and access datasets on each
of the five platforms, connect to common federal data APIs programmatically,
and navigate the access approval processes without slowing down their project.

Platform Coverage: All five platforms with platform-specific acquisition
patterns. Deepest coverage on Advana data catalog and Palantir Ontology.
Code Examples: FPDS-NG API data pull; Advana catalog query; Databricks Unity
Catalog metadata scan; Palantir Foundry dataset load.

CHAPTER 04: Data Wrangling and Cleaning
Directory: chapters/04-data-wrangling/
──────────────────────────────────────────────────────────────
Scope:
Government data is messy in specific ways — inconsistent agency codes, fiscal
year date formats that don't match calendar year conventions, NAICS/PSC codes
that change meaning across contract years, legacy system exports with 1980s
encoding artifacts. This chapter covers the full wrangling toolkit with
examples drawn from actual government data patterns.

Topics:
- Pandas fundamentals for wrangling: the operations practitioners actually use
  (merge, groupby, pivot, melt, str.extract) versus what textbooks teach
- Handling government-specific data formats: fiscal year (FY) dates, DoD
  DODAAC codes, contract numbers (PIID format), UEI/DUNS vendor identifiers,
  NAICS/PSC/SIC codes, agency codes (TAS, BEA)
- Missing data strategies: when to impute, when to drop, when to escalate to
  the data steward — with specific guidance for federal datasets where missing
  data often has a meaning (undisclosed, restricted, pending)
- Deduplication: entity resolution on vendor names, address standardization,
  identifying duplicate contract modifications
- Data type enforcement and schema validation: using pandera and Great
  Expectations for data quality checks in government pipelines
- Large-scale wrangling on platforms: PySpark on Databricks for multi-gigabyte
  federal datasets; Palantir Foundry transforms; Advana pipeline patterns
- Qlik data load scripting: QVD optimization, incremental loading, set analysis
  for fiscal year period handling
- Data lineage documentation: tracking transformations for audit purposes

Learning Objectives:
After this chapter, the practitioner can clean and transform common federal
government datasets using Python/pandas and platform-native tools, validate
data quality programmatically, and document their transformations to satisfy
audit requirements.

Platform Coverage: All five platforms. PySpark/Databricks and Palantir
Transforms in depth for large-scale wrangling.
Code Examples: Federal contract data cleaner (FPDS-NG); PySpark pipeline for
DoD financial data; Palantir Transform for ontology-linked entity resolution.

CHAPTER 05: Exploratory Data Analysis
Directory: chapters/05-exploratory-analysis/
──────────────────────────────────────────────────────────────
Scope:
EDA in a federal context has constraints commercial data science doesn't:
you often cannot share your findings in public notebooks, your visualizations
may need to be classified, and your "interesting finding" might be something
your government customer is legally prohibited from acting on without a formal
analysis report. This chapter covers EDA techniques and the federal context
they operate in.

Topics:
- Descriptive statistics with government data: what summary statistics reveal
  about procurement data, personnel data, readiness data, financial data — and
  what they hide
- Visualization for federal audiences: choosing charts for non-technical
  government stakeholders; what works in briefings to flag officers and SES
  officials; Qlik visualization patterns for executive dashboards
- Feature engineering for government datasets: deriving useful features from
  contract data (days-to-award, modification frequency, recompete history);
  time series features from financial data; categorical encodings for
  agency/command hierarchies
- Outlier detection in government context: distinguishing anomalies from
  legitimate exceptions (supplemental appropriations, emergency contract
  actions, directed buys)
- Correlation and causation in policy analysis: pitfalls unique to government
  data (Simpson's paradox in agency-level rollups, survivorship bias in
  contract award analysis)
- EDA on each platform: Databricks notebooks with plotly/matplotlib; Palantir
  Foundry Analysis objects; Qlik sheet design for exploratory work; Advana
  workspace-based analysis
- Documenting EDA for knowledge transfer: markdown notebooks, version-
  controlled analysis outputs, briefing-ready export patterns

Learning Objectives:
After this chapter, the practitioner can run rigorous EDA on federal government
datasets, produce visualizations appropriate for government audiences, and
document their analytical process to support peer review and audit.

Platform Coverage: All five platforms. Qlik sheet design and Palantir Analysis
in depth.
Code Examples: Government contract EDA notebook; Qlik set analysis for fiscal
year comparisons; Databricks profiling with ydata-profiling.

CHAPTER 06: Supervised Machine Learning
Directory: chapters/06-supervised-ml/
──────────────────────────────────────────────────────────────
Scope:
This chapter covers supervised learning from the perspective of a practitioner
building models for actual government use cases — fraud detection in contracting,
readiness prediction, classification of acquisition anomalies, budget forecasting.
The focus is on model selection judgment, evaluation rigor, and what it takes to
get a model approved for government use.

Topics:
- Classification and regression on government datasets: concrete use cases
  (contract award prediction, audit risk scoring, supply chain anomaly
  detection, personnel readiness classification)
- Algorithm selection in practice: when to use logistic regression vs. gradient
  boosting vs. random forest on federal datasets; the performance/explainability
  tradeoff in government contexts where decisions must be defensible
- Feature engineering for ML: what carries signal in procurement data,
  financial data, and readiness data
- Cross-validation and evaluation: confusion matrices, ROC/AUC, precision-
  recall tradeoffs for government decision use cases; evaluating against
  imbalanced datasets (rare events in fraud/anomaly detection)
- Model explainability requirements: SHAP values, LIME, and the DoD
  requirements for explainable AI (DoD AI Ethics Principles, algorithmic
  accountability)
- Hyperparameter tuning on platform: scikit-learn pipelines; Databricks
  Hyperopt; Palantir Foundry model training workflow
- MLflow experiment tracking: logging parameters, metrics, and artifacts for
  government deliverables; tracking experiments in Databricks
- scikit-learn on Databricks: distributed model training with pandas_udf and
  Spark ML
- Getting a model approved for use: ATO considerations for ML models, model
  cards, bias testing requirements per the DoD AI ethical principles

Learning Objectives:
After this chapter, the practitioner can build, evaluate, and document
supervised ML models for government use cases and understands the approval
requirements for deploying a model in a federal environment.

Platform Coverage: Databricks and Palantir AIP in depth for training workflows;
all five platforms for deployment patterns.
Code Examples: Contract award risk classifier (scikit-learn + MLflow); Spark ML
pipeline for large-scale government dataset; Palantir Foundry model training.

CHAPTER 07: Unsupervised Machine Learning
Directory: chapters/07-unsupervised-ml/
──────────────────────────────────────────────────────────────
Scope:
Unsupervised learning is underused on government programs because results are
harder to explain to program managers and contracting officers. This chapter
focuses on making unsupervised results interpretable and defensible for
government audiences — while covering the techniques that actually surface
useful patterns in federal data.

Topics:
- Clustering for government use cases: vendor behavior segmentation, entity
  grouping in procurement data, anomaly cluster identification in financial
  transactions, operational pattern analysis
- K-means vs. DBSCAN vs. hierarchical clustering: when to use each on
  government datasets; how cluster count selection translates to briefing
  narrative
- Dimensionality reduction: PCA for high-dimensional procurement data; UMAP
  and t-SNE for visualization; what reduction reveals about latent structure
  in agency-level data
- Anomaly detection: isolation forests, autoencoders, and one-class SVM for
  audit flag generation; the difference between statistical anomaly and
  operational significance in a DoD context
- Topic modeling for government text data: LDA on contract solicitation
  documents, safety reports, maintenance records; BERT embeddings for more
  modern approaches
- Association rules for procurement analysis: finding unexpected co-occurrence
  patterns in multi-award contracts, vendor relationships, agency purchasing
  patterns
- Platform-specific implementations: Databricks MLlib for large-scale
  clustering; Palantir Foundry for ontology-aware entity grouping; Qlik for
  visualization of cluster outputs

Learning Objectives:
After this chapter, the practitioner can apply unsupervised learning to
government datasets, explain cluster and anomaly results to non-technical
government stakeholders, and document findings in a format that supports
further analysis and formal reporting.

Platform Coverage: Databricks MLlib in depth; Palantir Foundry entity
resolution and grouping; Qlik for cluster visualization.
Code Examples: Vendor segmentation with K-means on FPDS data; anomaly
detection pipeline for financial transactions; LDA on contract solicitations.

CHAPTER 08: Deep Learning and Neural Networks
Directory: chapters/08-deep-learning/
──────────────────────────────────────────────────────────────
Scope:
Deep learning on federal platforms means working around GPU availability
constraints, long approval processes for framework deployment, and the reality
that most government data science problems do not require deep learning. This
chapter teaches practitioners to know when deep learning is warranted and how to
execute it on the platforms that support it.

Topics:
- When to use deep learning and when not to: the honest evaluation of whether
  a government use case benefits from neural networks vs. gradient boosting;
  compute costs on government platforms
- Neural network fundamentals through a practical lens: what a practitioner
  needs to understand vs. what is academic background
- Computer vision for government applications: satellite imagery analysis,
  document digitization (OCR for legacy paper records), facility inspection
  image classification — with platform-specific implementation
- NLP in government context: classification of FOIA requests, sentiment
  analysis on contractor performance reports, entity extraction from
  contracting documents
- Deep learning frameworks on federal platforms: PyTorch and TensorFlow on
  Databricks with GPU-enabled clusters; Palantir AIP for LLM integration;
  availability constraints on Advana and Navy Jupiter
- Transfer learning: fine-tuning pre-trained models on government domain data
  (domain-specific text, government document formats, geospatial data)
- Training infrastructure on Databricks: GPU cluster configuration, distributed
  training with Horovod/DeepSpeed, cost management
- Model compression for deployment in resource-constrained government
  environments: quantization, pruning, ONNX export

Learning Objectives:
After this chapter, the practitioner can evaluate whether a use case warrants
deep learning, configure GPU compute on Databricks, implement computer vision
or NLP pipelines for government-relevant tasks, and export models for
deployment.

Platform Coverage: Databricks GPU compute in depth; Palantir AIP for LLM
integration; coverage of constraints on Advana/Jupiter for deep learning.
Code Examples: Document classification with fine-tuned BERT; satellite image
classifier on Databricks GPU cluster; ONNX model export for edge deployment.

CHAPTER 12: Ethics, Governance, and Compliance
Directory: chapters/12-ethics-governance/
──────────────────────────────────────────────────────────────
Scope:
Ethics and compliance in federal AI/ML is not a philosophical exercise — it
is a legal and contractual requirement with specific standards, specific
certifications, and specific people whose job it is to enforce them. This
chapter covers what practitioners actually have to do to satisfy those
requirements, not just what they mean in theory.

Topics:
- DoD AI Ethics Principles (2020): the five principles (responsible,
  equitable, traceable, reliable, governable) and what each means for a
  practitioner building a model — not the policy language, the implementation
  checklist
- Algorithmic accountability in federal programs: who is responsible when a
  model produces a bad outcome; what documentation is required; how to write
  a model card that satisfies a DoD program office
- Bias testing requirements: what protected categories matter in federal
  contexts; fairness metrics (demographic parity, equalized odds) and when
  the government requires you to test for them; tools (Fairlearn, AIF360)

CHAPTER 13: Advanced Topics — GenAI, RAG, and LLMs on Federal Platforms
Directory: chapters/13-advanced-topics/
──────────────────────────────────────────────────────────────
Scope:
As of 2026, generative AI has arrived in the federal space — not as a
research curiosity, but as a deployed capability on multiple DoD programs.
Palantir AIP is running in production on DoD contracts. Databricks has DBRX
and full LLM serving capabilities. The question practitioners face is not
whether to use LLMs, but how to use them responsibly within federal
constraints. This chapter covers the current state of GenAI in the federal
space with the specificity that practitioners need.

Topics:
- The GenAI landscape for federal practitioners (2025-2026): what has been
  authorized, what is in pilot, what remains prohibited — with specific
  platform context (which capabilities are IL4/IL5 authorized)

  researcher level); what fine-tuning vs. RAG vs. prompt engineering each
  accomplishes; when to use which on a government program
- Retrieval-Augmented Generation (RAG) for government use cases:
  grounding LLM responses in authoritative government documents (policy,
  doctrine, contract data, intelligence reports at unclassified level);
  RAG architecture patterns; vector databases on federal platforms
- Palantir AIP: the AI Platform layer built on Foundry; AIP Logic for

- Databricks AI and GenAI: Mosaic AI (formerly Databricks ML); LLM fine-tuning
  on Databricks; Vector Search for RAG; AI Playground; mlflow.langchain
  integration; DBRX and open-source LLM options for air-gapped environments
- LLM deployment on IL4/IL5 environments: the challenge of air-gapped LLM
  deployment; open-source models (Llama, Mistral, Falcon) as alternatives to
  cloud API LLMs; on-premises inference with Ollama and similar tools
- Prompt engineering for government work: effective prompt patterns for document
  analysis, contract review, report generation, and question-answering over
  government datasets; adversarial prompting and defense
- Evaluation of LLM outputs in federal contexts: RAGAS, factuality scoring,
  hallucination detection; why evaluation is more critical when an LLM is
  advising on policy or operational decisions
- AI governance for GenAI in DoD: the DoD Generative AI Policy (2023);
  CDAO guidance on responsible use; contractor obligations; what you cannot
  automate with GenAI (decision authorities, personnel actions, classified

Learning Objectives:

SECTION 6: CONTENT QUALITY STANDARDS
─────────────────────────────────────────────────────────────────────────────

These standards are the immutable quality bar for all content. They are
defined in docs/STYLE_GUIDE.md and docs/CHAPTER_WRITING_SPEC.md and cannot
be modified by content-writing agents.

6.1 Voice and Writing Standards
─────────────────────────────────
The reference voice is Amyn Porbanderwala's "Shrink-Wrap It: The GovCon
Productization Playbook." Content must match these characteristics:

REQUIRED:
- Open sections with concrete practitioner scenarios (not definitions,
  not thesis statements)
- Use second person ("you") addressing the practitioner directly
- State opinions plainly without hedging in main prose
- Use specific numbers, names, and data points
- Vary sentence length deliberately (short punches after long setups)
- Raise objections before answering them
- End chapters with three-block close: one thing to remember / Monday action /
  what comes next
- Put caveats in Note: or Sanity check: blocks, not in prose

PROHIBITED (these are AI-writing tells and will fail QA):
- Banned words: delve, leverage (as verb), landscape (as in "the landscape"),
  navigate (as in "navigate challenges"), crucial, robust, seamless, empower,
  harness (as in "harness the power"), unlock, game-changer, paradigm shift,
  synergy, comprehensive, foster, facilitate, bolster, spearhead
- Banned structural patterns: uniform paragraph length, balanced pros/cons
  without a point of view, "Firstly/Secondly/Thirdly" enumeration, wrapping
  every section in intro-body-summary sandwich, opening with "In this chapter
  we will explore...", using bullet points for everything

6.2 Technical Accuracy Standards
──────────────────────────────────
- All technical claims must be based on publicly verifiable information
- No invented API endpoints, made-up platform features, or fabricated
  compliance requirements
- Platform-specific code must use actual API conventions for each platform
  (or be clearly marked as pseudocode where the API is proprietary)
- Security and compliance claims must cite the correct standards (e.g., when
  saying a platform has IL4 authorization, this must be publicly documented)
- Government data use cases must be realistic — not Iris dataset, not
  generic retail data, not Titanic survival prediction

6.3 Code Quality Standards
───────────────────────────
- Python 3.11+ compatible syntax required
- Code examples must be syntactically correct and runnable (or clearly marked
  as pseudocode/illustrative for platform-specific APIs)
- All code includes comments explaining non-obvious lines
- All code includes sample output where it aids understanding
- No truncation with "# ... more code here" — examples are complete
- No secrets, credentials, or PII in any code example
- Code examples use realistic government/defense data scenarios (not toy
  datasets where a real scenario exists)

6.4 Completeness Standards
───────────────────────────
Per chapter minimum requirements:
- 2500+ words of substantive prose (not counting code blocks)
- 3+ working code examples (Python required, R optional)
- 1+ Mermaid diagram
- 3+ hands-on exercises with solutions
- Coverage of at least 3 of 5 platforms where the topic is relevant
- Zero TODO/TBD/[insert]/placeholder markers
- Chapter opens with a concrete practitioner scenario
- Chapter ends with the three-block close
- Cross-references to related chapters where appropriate

- Full-length data science curriculum for beginners — this handbook assumes
  practitioners who already know Python and basic ML concepts
- Business intelligence licensing and procurement (beyond what a practitioner
  needs to understand access)
- Database administration (practitioners consume data, they do not admin it)
- Network engineering or infrastructure administration
- Formal academic treatment of statistical theory — the handbook is applied,
  not theoretical
- Reinforcement learning (too narrow/specialized for this audience)
- Quantum computing (too early-stage for federal operational context)
- Blockchain / distributed ledger applications

10.3 Delivery Formats Not In Scope
─────────────────────────────────────
- PDF or print-ready formatting — the handbook is markdown/web-first
- Video content, screencasts, or interactive tutorials
- SCORM packages or LMS-compatible formats
- Translated versions (English only for this version)
- Printed textbook layout with ISBN

10.4 Infrastructure Not In Scope
──────────────────────────────────
- Production deployment of the handbook itself to a government-authorized
  hosting environment — this is a repository, not a deployed application
- Formal Authority to Operate (ATO) for the handbook repository
- ITAR/EAR compliance review (content is unclassified public information)
- Formal Section 508 accessibility compliance audit (though markdown should
  be accessible by default)
- Integration with a Learning Management System (LMS)
