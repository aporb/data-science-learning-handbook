## Solution 1: The Null Pattern Hunt

### Step 1: Overall Null Rate

```python
from chapters.code_examples.python.statistical_profiling import generate_synthetic_don_dataset

df = generate_synthetic_don_dataset(n_rows=50_000)

overall_null_rate = df["completion_date"].isnull().mean() * 100
print(f"Overall null rate for completion_date: {overall_null_rate:.1f}%")
# Expected output: approximately 8.0%
```

### Step 2: Null Rate by Source System

```python
from statistical_profiling import analyze_null_patterns

null_pattern = analyze_null_patterns(
    df,
    null_col="completion_date",
    group_cols=["source_system"]
)
print(null_pattern.to_string())
```

**Expected output:** The synthetic dataset injects null completion dates at a higher rate for the SAMS-E source system (documented in the dataset generator's docstring). You should see null rates of roughly:
- SAMS-E: ~12–15%
- SMCS: ~4–6%
- DPAS: ~4–6%
- NALCOMIS: ~4–6%

The SAMS-E rate is roughly 2x or more the other systems — `analyze_null_patterns()` should flag this as clustered.

### Step 3: Finding Statement

> "The `completion_date` field has an overall null rate of 7.9% (approximately 4,000 of 50,000 records). These nulls are not randomly distributed: SAMS-E records account for approximately 60% of all null completion dates despite contributing only 50% of records overall. SAMS-E has a null rate approximately 2.4x higher than the other three source systems combined. Before proceeding with any analysis that uses completion_date — including time-to-completion, overdue metrics, or workload models — we recommend: (1) contacting the SAMS-E data steward to understand whether this is a known pipeline gap, and (2) excluding SAMS-E records from time-to-completion analysis or documenting the exclusion explicitly as a limitation."

### Step 4: Random vs. Clustered Handling

**If nulls are random:** You can impute (e.g., median imputation or model-based imputation) and proceed, documenting the imputation method. The 8% rate is acceptable for most analyses.

**If nulls are clustered in SAMS-E:** You cannot impute safely, because SAMS-E records may be systematically different from other records in ways you cannot observe — the missing data is missing for a reason. Options: (a) exclude SAMS-E records from time-to-completion analysis and note the exclusion; (b) treat SAMS-E records as a separate population; (c) escalate to the data steward to get the missing completions from the source system.

## Solution 2: The Impossible Date Problem

### Step 1: Count Negative Values

```python
neg_mask = df["days_elapsed"] < 0
print(f"Negative days_elapsed: {neg_mask.sum():,} ({neg_mask.mean() * 100:.1f}%)")
# Expected: approximately 600 records, ~1.2% of dataset
```

### Step 2: Domain Range Validation

```python
from statistical_profiling import validate_domain_ranges

rules = {
    "days_elapsed": {"min": 0, "max": 1825},  # 0 days minimum; 5-year max
}
violations = validate_domain_ranges(df, rules)
print(violations)
```

The output should show approximately 600 violations of the `days_elapsed >= 0` rule. The `validate_domain_ranges()` output includes count, percentage, and sample values.

### Step 3: Three Options Analysis

| Option | What You Lose | What You Gain | When to Choose |
|---|---|---|---|
| A: Remove | Approximately 1.2% of records; if these records have a non-random pattern (e.g., specific ship classes or event types), you introduce bias | Clean data for time-to-completion models | When the 1.2% is truly random and the analysis is sensitive to negative values |
| B: Set to 0 | The true distribution of short-duration jobs (0 = "same day completion" becomes indistinguishable from the corrected error records) | Preserves record count | When you have strong reason to believe the negative is a systematic recording error that happened at completion (completion recorded before start was finalized) |
| C: Flag and keep | Model needs to handle the flag explicitly as a feature or exclusion | Preserves all records; makes the quality issue visible in the analysis; enables per-analysis decision-making | When you don't know whether the negatives are errors or signal, and you need to preserve optionality |

**Option C is usually correct for the first pass.** You can always remove or impute later. You cannot recover from removing records if it turns out they were signal.

### Step 4: Response to "Just Remove Them"

"Before removing them, we should check whether the negative records cluster in any specific ship class, event type, or source system. If they're random, removal is fine. If they cluster — say, all negative records are from one shipyard or one event type — then removing them would bias our analysis and potentially make us blind to whatever operating condition produces them."

### Code Example with Documentation

```python
# Decision: Flag negative values as data quality issues, keep in dataset
# Rationale: 1.2% of records, likely recording errors (completion date entered
# before start date was updated), but distribution across ship classes is roughly
# uniform (checked: chi-squared p=0.43). No evidence of systematic bias.
# Per data steward contact on 2024-03-22: SAMS-E has a known race condition
# between completion entry and start date update. These can safely be set to 0
# for time-to-completion analysis after confirming with the steward.

df["days_elapsed_dq_flag"] = df["days_elapsed"] < 0
df["days_elapsed_analysis"] = df["days_elapsed"].clip(lower=0)

# Log this decision
print(f"Flagged {df['days_elapsed_dq_flag'].sum()} records with negative days_elapsed")
print("Using days_elapsed_analysis (clipped at 0) for model features")
print("Using days_elapsed_dq_flag as a potential feature or for exclusion per analysis need")
```

### Step 3: Model Failure Modes

A staffing forecast model that ignores FY seasonality would:
- **Predict too low** for September — it would use the average month's staffing need for September, but actual demand is 3.5x higher
- **Predict too high** for October and November — the post-September slowdown (backlog cleared, new FY not yet ramped) would look like unexplained underperformance against a flat baseline
- The model would systematically miss the peak demand period that most strains maintenance workforce capacity

### Step 4: Features to Capture FY Seasonality

1. **`is_fy_end_month`** (binary): 1 if the record falls in September, 0 otherwise. Captures the step-change in September demand.

2. **`fy_month_index`** (1–12, where 1=October): Captures the gradual ramp through the year — many maintenance programs accelerate toward FY end starting in July, so a linear month index helps the model see that acceleration.

Optional: **`days_to_fy_end`** (continuous): Number of days until September 30 of the current fiscal year. Captures the urgency gradient more granularly than a month index.

## Solution 4: Procurement Threshold Analysis

### Step 1 and 2: Band Analysis

```python
import pandas as pd
import numpy as np

threshold = 250_000
window = 0.05

below_band = (df["contract_value"] >= threshold * (1 - window)) & (df["contract_value"] < threshold)
above_band = (df["contract_value"] >= threshold) & (df["contract_value"] < threshold * (1 + window))

n_below = below_band.sum()
n_above = above_band.sum()
total_band = n_below + n_above

print(f"Just below $250K ({threshold*(1-window):,.0f}–{threshold:,.0f}): {n_below:,} records")
print(f"Just above $250K ({threshold:,.0f}–{threshold*(1+window):,.0f}): {n_above:,} records")
print(f"Ratio below / (below + above): {n_below/max(total_band,1):.1%}")
```

**Expected interpretation:** Under a log-normal distribution with the synthetic dataset's parameters, you would expect roughly a 45/55 or 50/50 split between the below and above bands (the log-normal curve is nearly symmetric at any specific point). A ratio significantly above 70/30 below-to-above would indicate threshold-gaming. The synthetic dataset uses a log-normal distribution without injecting threshold gaming, so you should see roughly equal counts in both bands.

### Step 3: Vendor Pattern Query

# Find vendors with 5+ contracts in the $237,500–$250,000 range within a fiscal year

### Step 4: Interpretation with Domain Context

The program officer's point is valid and important. Contracting officers routinely use simplified acquisition procedures for contracts below the $250,000 threshold — it's faster, requires less documentation, and is appropriate for the acquisition. So you would expect clustering below that threshold.

What changes your interpretation is the *rate* of clustering, not the fact of it. If 55% of contracts in a 10% band around $250,000 fall below the threshold, that's expected. If 92% do — across many vendors in the same fiscal year — that's statistically unusual even given legitimate simplified acquisition use.

The threshold analysis is a screening tool, not a finding. It surfaces candidates for human review, not conclusions about fraud.

### Step 3: HIGH Severity Finding Analysis

The expected HIGH severity findings from this dataset are:

**Finding 1: Negative days_elapsed (~600 records, ~1.2%)**
- Plain English: Some work orders show negative elapsed time — the completion date appears before the start date in the system.
- Model impact: A predictive maintenance model using `days_elapsed` as a feature would learn from nonsensical negative values. These records would cluster in the "fast completion" prediction bucket even though they represent data errors.
- Remediation: Contact SAMS-E data steward to confirm this is the known race condition. Apply `clip(lower=0)` in the Silver transform pipeline, flag in a `days_elapsed_dq_flag` column. Document in model card.

**Finding 2: Negative contract_value (~0.8% of records)**
- Plain English: Some maintenance contracts show negative dollar amounts.
- Model impact: A cost prediction model trained on this data would include negative cost targets, producing nonsensical predictions. A maintenance budget forecast would understate actual costs.
- Remediation: Negative contract values should be reviewed against original source records. If they represent corrections (deobligations), they should be handled as a separate record type. If they are data entry errors, set to `NULL` in the Silver pipeline and exclude from cost models.

### Step 4: Sample Data Quality Memo

---

**MEMORANDUM**

**TO:** [Program Manager Name]
**FROM:** [Analyst Name]
**SUBJECT:** Data Quality Assessment — DoN Maintenance Events Dataset, FY2020–FY2024
**DATE:** [Date]

**Dataset Overview**

The DoN Maintenance Events dataset contains 51,500 records (including approximately 1,500 duplicate records) spanning October 2019 through September 2024. The dataset covers four source systems: SAMS-E (50% of records), SMCS (20%), DPAS (18%), and NALCOMIS (12%). It includes work order metadata, event types, hull numbers, elapsed days, and contract values.

**Significant Findings**

*Finding 1 — Negative elapsed time (HIGH severity):* Approximately 600 records show negative values in the `days_elapsed` field, meaning the recorded completion date precedes the recorded start date. This represents 1.2% of the dataset. These records appear distributed across source systems and ship classes in roughly equal proportions, suggesting a systematic recording issue rather than a single data pipeline failure. We recommend contacting the SAMS-E data steward to confirm the root cause before using `days_elapsed` in any model or metric.

*Finding 2 — Null completion dates clustered in SAMS-E (MEDIUM severity):* The overall null rate for `completion_date` is 7.9%. However, SAMS-E records have a null rate approximately 2.4x higher than the other source systems. This clustering indicates a data collection gap specific to SAMS-E, not a random sample issue. Time-to-completion metrics and any model trained on completion timing will be biased if SAMS-E's higher missing rate is not accounted for.

*Finding 3 — Fiscal year end spike (MEDIUM severity):* Work order volume in September is approximately 3.5x the monthly average for the rest of the year. This is consistent with known DoD fiscal year budget execution patterns. Any staffing model, workload forecast, or capacity planning analysis must account for this seasonality explicitly; otherwise it will systematically underpredict September demand.

**Recommended Actions**

The two HIGH and MEDIUM findings above should be resolved before this dataset is used as a training input for any predictive model. Specifically: (1) the negative elapsed time issue requires a data steward consultation and a documented remediation decision; (2) the SAMS-E null pattern requires either an exclusion with documentation or a stew confirmation that imputation is appropriate. The FY seasonality finding should be addressed in model design rather than data cleaning.

This dataset is appropriate for descriptive analysis and baseline characterization with the limitations noted above. It is not appropriate as a training input for a predictive maintenance model until the HIGH severity findings are resolved.

## Solution 6: Platform Selection Scenario

1. **Collibra catalog (Advana) first.** Collibra documents field definitions, known issues, lineage, and the data steward — exactly what you need before writing any code. This takes 20 minutes and may save you 2 days.

2. **Databricks (Advana tenant).** `profile_dataframe()` from `01_statistical_profiling.py` runs across all 47 columns in one pass, producing null rates, cardinality, and dtype consistency systematically.

3. **Qlik Sense (Advana).** Three clicks to build a histogram of obligation_amount on an Advana Qlik app. If a Qlik app already exists for this dataset (check first), you can have a histogram in under two minutes without writing any code.

4. **Qlik Sense or Databricks.** For comparing two group distributions, both work. In Qlik, select Agency A and observe the distribution chart for contract values; deselect and select Agency B. In Databricks, `df.groupby("agency").describe()` gives you the same answer in code. Use Qlik if you're already there; use Databricks if you need the comparison documented in reproducible code.

5. **Qlik Sense first, then Databricks.** The associative model is exactly designed for this: select "top 1% obligation amount" and "Sole Source competition" simultaneously, observe what characteristics turn green across all other dimensions. Then translate the finding to a Databricks notebook filter for documentation.

6. **Databricks.** Qlik apps are not version-controlled in a way your team lead can review as code. Databricks notebooks in Repos are. This is the documentation platform.

**Conclusion:** No single platform handles all six tasks equally well. The efficient approach is: Collibra first (task 1), Qlik for rapid visual exploration (tasks 3–5), Databricks for systematic profiling and reproducible documentation (tasks 2, 6). This is how the chapter describes it — use Qlik for hypotheses you already have, Databricks for discovery and documentation.
